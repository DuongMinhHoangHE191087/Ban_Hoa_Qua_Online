Phân Trang, DTO & Chỉ Mục

Cẩm nang chi tiết về lý thuyết, sơ đồ hoạt động và hướng dẫn tích hợp công nghệ phân trang tối ưu.

1

Data Transfer Object (DTO)

Khi thực hiện câu lệnh SELECT * hoặc SELECT p.* trên bảng sản phẩm, cơ sở dữ liệu sẽ phải tải toàn bộ thông tin của dòng bao gồm cột description kiểu dữ liệu NVARCHAR(MAX).

Vấn đề: Cột dữ liệu lớn chiếm dụng cực kỳ nhiều bộ nhớ đệm (Buffer Pool) của Database, tăng I/O đĩa và băng thông truyền dữ liệu trên mạng. Nếu danh sách hiển thị có 20 sản phẩm, việc giữ 20 chuỗi mô tả dài trong RAM của Web Server (JVM) sẽ nhanh chóng gây phân mảnh bộ nhớ và làm chậm Garbage Collector (GC).

Giải pháp của chúng ta:

Chúng ta tạo lớp ProductDTO kế thừa từ Product để đảm bảo tính tương thích ngược hoàn toàn với giao diện JSP JSTL. Trong các truy vấn danh sách, ta chỉ SELECT các trường cần thiết và thay thế mô tả bằng giá trị CAST(NULL AS NVARCHAR(MAX)). 

// Câu lệnh SQL SELECT DTO tối ưu:
SELECT p.product_id, p.name, p.price, p.unit, 
       CAST(NULL AS NVARCHAR(MAX)) AS description, 
       p.status, p.approval_status ...
FROM products p

Lý Thuyết Phân Trang (Offset vs Keyset)

Đặc trưng OFFSET / FETCH Pagination KEYSET / CURSOR Pagination
Cú pháp SQL OFFSET ? ROWS FETCH NEXT ? ROWS ONLY WHERE id < ? ORDER BY id DESC FETCH NEXT ? ROWS
Hiệu năng trang sâu Kém (O(N) - Phải quét và bỏ qua N dòng đầu) Cực nhanh (O(log N) - Nhảy trực tiếp bằng chỉ mục)
Tránh trùng lặp Dễ bị trùng/sót bản ghi khi có thêm mới/xóa dữ liệu Tuyệt đối chính xác (Không phụ thuộc số dòng đã quét)
Ứng dụng Thanh chuyển trang truyền thống (Trang 1, 2, 3...) Tải thêm dữ liệu (Infinite Scroll), API dữ liệu lớn

1. Offset-Based Pagination (Phân trang Offset)

Database tìm kiếm dòng bắt đầu dựa trên số dòng cần bỏ qua. Ví dụ: muốn xem trang 5 với kích thước 10 dòng, database phải quét qua 40 dòng đầu tiên rồi lấy 10 dòng tiếp theo. Nếu xem trang 10,000, database phải đọc 100,000 dòng rồi vứt đi, cực kỳ lãng phí.

2. Keyset/Cursor-Based Pagination (Phân trang Keyset)

Database định vị dòng bắt đầu dựa trên giá trị của khóa của bản ghi cuối cùng ở trang trước. Ví dụ: *"Lấy 10 sản phẩm có ID nhỏ hơn ID sản phẩm cuối cùng của trang trước là 42"*. Nhờ chỉ mục trên khóa chính, database thực hiện phép so sánh trực tiếp, bỏ qua việc quét tuần tự. 

// Query Keyset Pagination:
SELECT p.product_id, p.name FROM products p
WHERE p.status = 'ACTIVE' AND p.product_id < ?
ORDER BY p.product_id DESC
OFFSET 0 ROWS FETCH NEXT ? ROWS ONLY
3

Chỉ Mục Composite (Composite Index)

Mọi truy vấn lọc sản phẩm hiển thị trên trang chủ và trang danh sách đều có điều kiện: WHERE status = 'ACTIVE' AND approval_status = 'APPROVED' và được sắp xếp giảm dần theo product_id DESC.

Khi chưa có chỉ mục, hệ quản trị cơ sở dữ liệu (SQL Server) phải quét toàn bộ bảng (Table Scan/Clustered Index Scan) để lọc và sắp xếp. Bằng cách thêm chỉ mục ghép: 

CREATE INDEX IX_products_status_approval_product_id 
ON products(status, approval_status, product_id DESC);

Dữ liệu trong chỉ mục đã được phân nhóm sẵn theo trạng thái hoạt động, phê duyệt và sắp xếp sẵn thứ tự ID giảm dần. Truy xuất lúc này trở thành phép tìm kiếm nhị phân (Index Seek) cực kỳ nhanh chóng (dưới 1ms).
