[CheckoutServlet L226] ownerId hardcode = 1 — Đơn hàng luôn gán cho owner sai

📄 File: CheckoutServlet.java📍 Dòng 226⚠️ Impact: Toàn bộ đơn hàng sẽ hiển thị sai chủ shop

Biến ownerId được khởi tạo mặc định là 1. Logic lấy owner_id từ DB nằm trong khối if (firstVariant != null) nhưng query này mở Connection riêng biệt trước transaction — nếu firstVariant == null thì ownerId vẫn là 1 (hardcode), gây gán sai owner.

ProductVariant firstVariant = productVariantDAO.findById(checkoutItems.get(0).getVariantId());
if (firstVariant != null) {
    // Chỉ cập nhật khi firstVariant != null — nếu null thì ownerId = 1 !!
    try (Connection conn = ...) { ... }
}
FIX: Throw exception ngay nếu firstVariant == null thay vì fallback về ownerId = 1. Đồng thời, việc truy vấn owner_id nên nằm trong DAO (ProductDAO.getOwnerId(productId)) và có thể gộp vào transaction chính.

[CheckoutServlet L213-220] Double DB Query cho giá — Race condition & N+1 problem

📄 File: CheckoutServlet.java📍 Dòng 193-220⚠️ Impact: N DB calls extra, giá có thể thay đổi giữa 2 vòng lặp

productVariantDAO.findById() được gọi 2 lần riêng biệt cho mỗi item: lần 1 ở vòng kiểm tra stock (L194-200), lần 2 ở vòng tính tiền (L215). Với 10 items = 20 DB round-trips. Nghiêm trọng hơn: giá có thể thay đổi giữa 2 lần gọi này (dù hiếm).

// Vòng 1: Kiểm tra stock — query findById cho mỗi item
for (CartItem item : checkoutItems) {
    ProductVariant variant = productVariantDAO.findById(item.getVariantId()); // QUERY 1
    ...
}
// Vòng 2: Tính tiền — query findById LẠI cho mỗi item
for (CartItem item : checkoutItems) {
    ProductVariant variant = productVariantDAO.findById(item.getVariantId()); // QUERY 2 (DUPLICATE)
    ...
}
FIX: Load tất cả variants trong 1 query bằng WHERE variant_id IN (...), lưu vào Map<Integer, ProductVariant> variantMap, dùng lại cho cả 2 vòng lặp.

[CheckoutServlet L270-271] Lần 3 query findById TRONG transaction — Connection leak tiềm ẩn

📄 File: CheckoutServlet.java📍 Dòng 270-271⚠️ Impact: Lần gọi thứ 3 mở connection mới trong khi transaction đang mở

Bên trong transaction (conn.setAutoCommit(false)), vòng lặp insert order_items gọi productVariantDAO.findById() lần thứ 3 cho mỗi item. ProductVariantDAO tự mở connection mới từ pool — đây là connection khác với conn đang giữ transaction, gây tối thiểu 3 connections mở đồng thời cho mỗi item và có thể deadlock trên SQL Server (SELECT ... trong khi UPDATE đang lock row).

// BÊN TRONG TRANSACTION
try (PreparedStatement itemPs = conn.prepareStatement(...)) {
    for (CartItem item : checkoutItems) {
        ProductVariant variant = productVariantDAO.findById(...); // ← Connection THỨ 3 từ pool!
        ...
    }
}
FIX: Cache variantMap trước khi mở transaction (giải quyết cả Bug #2). Không gọi DAO tự quản lý connection bên trong khối transaction thủ công.
