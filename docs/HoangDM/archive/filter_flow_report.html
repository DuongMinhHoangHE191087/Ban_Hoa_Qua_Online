Filter Report

Luồng Filter xử lý request trong Ban Hoa Qua Online

Report này đọc theo web/WEB-INF/web.xml và các file trong src/java/filter/. Mục tiêu là giải thích vì sao filter chạy, chạy theo thứ tự nào, và mỗi filter dùng để làm gì.

Điểm cốt lõi

Filter không tự chạy vì tên file. Tomcat chỉ gọi filter khi request khớp <filter-mapping> trong web.xml. Mỗi filter xử lý xong phải gọi chain.doFilter(...) thì request mới đi tiếp. Nếu filter trả redirect, sendError hoặc ghi response rồi return, luồng sẽ dừng tại đó.

1. Filter chạy như thế nào

Với ứng dụng Servlet/JSP, request đi vào container trước, sau đó container ghép các filter phù hợp theo URL pattern. Thứ tự thực thi được quyết định bởi thứ tự khai báo trong web.xml, không phải thứ tự tên class.

Client
  -> Tomcat nhận request
  -> Match filter theo web.xml
  -> GlobalExceptionFilter
  -> SecurityHeadersFilter
  -> RateLimitFilter (chỉ auth/login, auth/register, api/*)
  -> EncodingFilter
  -> LoggingFilter
  -> SessionRestoreFilter
  -> CsrfFilter
  -> AuthFilter (chỉ URL bảo vệ)
  -> RoleFilter (chỉ URL bảo vệ)
  -> Servlet / JSP
  -> Response đi ngược trở lại qua chain

Khi một filter ném exception, GlobalExceptionFilter bọc bên ngoài sẽ bắt và chuẩn hóa phản hồi. Khi request hợp lệ, filter chỉ sửa request/response hoặc ghi log rồi chuyển tiếp bằng chain.doFilter(...).

3. Luồng thực tế khi chạy

Ví dụ 1: người dùng mở /shop/product bằng GET.

GlobalExceptionFilter
  -> SecurityHeadersFilter
  -> EncodingFilter
  -> LoggingFilter
  -> SessionRestoreFilter
  -> CsrfFilter (GET nên bỏ qua kiểm tra)
  -> AuthFilter (nếu URL nằm trong vùng bảo vệ)
  -> RoleFilter (nếu URL nằm trong vùng bảo vệ)
  -> Servlet / JSP

Ví dụ 2: người dùng submit form POST.

GlobalExceptionFilter
  -> SecurityHeadersFilter
  -> RateLimitFilter (nếu là /auth/* hoặc /api/*)
  -> EncodingFilter
  -> LoggingFilter
  -> SessionRestoreFilter
  -> CsrfFilter (kiểm tra token)
  -> AuthFilter
  -> RoleFilter
  -> Servlet xử lý dữ liệu

Điểm quan trọng là SessionRestoreFilter chạy trước AuthFilter, nên nếu cookie còn hợp lệ thì session có thể được phục hồi trước khi quyết định redirect login. RoleFilter chạy sau AuthFilter để chỉ kiểm tra quyền trên user đã xác thực.

4. Giải thích từng file
File Code gốc Giải thích ngắn
src/java/filter/GlobalExceptionFilter.java doFilter(...) Bọc toàn chain, gắn request id, bắt BusinessException, lỗi validation, lỗi security và lỗi hệ thống. Với AJAX thì trả JSON, với HTML thì redirect hoặc forward tới error page.
src/java/filter/SecurityHeadersFilter.java doFilter(...) Thêm header bảo mật cho mọi response. CSP đang cho phép một số CDN vì JSP hiện tại còn dùng nguồn ngoài và inline script/style.
src/java/filter/RateLimitFilter.java doFilter(...) Giới hạn số request theo IP trong cửa sổ 60 giây. Có bucket riêng cho auth và API, kèm cơ chế dọn entry cũ để tránh phình bộ nhớ.
src/java/filter/EncodingFilter.java doFilter(...) Đặt UTF-8 cho cả request và response. Filter này phải đi rất sớm để các servlet đọc tham số đúng mã hóa.
src/java/filter/LoggingFilter.java doFilter(...) Đo thời gian xử lý và log status cuối cùng. Vì log nằm trong finally, nó vẫn ghi được dù phía dưới có lỗi đã được xử lý.
src/java/filter/SessionRestoreFilter.java doFilter(...) Phục hồi session từ access token trước, rồi refresh token nếu cần. Bỏ qua static assets để không query DB vô ích.
src/java/filter/CsrfFilter.java doFilter(...) Tạo token CSRF trong session nếu chưa có, kiểm tra POST bằng form field hoặc header, và cho phép một số ngoại lệ đã thiết kế rõ ràng.
src/java/filter/AuthFilter.java doFilter(...) Chặn URL cần đăng nhập. Nếu session chưa có thì thử access token và refresh token. Nếu là AJAX thì trả 401 JSON, nếu là trình duyệt thì redirect về login.
src/java/filter/RoleFilter.java doFilter(...) Kiểm tra quyền theo role và theo ngữ cảnh nghiệp vụ. Nó còn đồng bộ user từ DB để tránh session bị stale khi role/status thay đổi.
5. Những điểm cần nhớ khi sửa filter
  • Thứ tự rất quan trọng: nếu đổi vị trí trong web.xml, logic bảo mật có thể bị hỏng hoặc response bị commit quá sớm.
  • Filter chỉ chặn khi cần: không nên áp dụng kiểm tra nặng cho static assets, vì sẽ làm chậm mọi request.
  • AJAX và form phải được xử lý khác nhau: các filter bảo mật hiện tại đã tách JSON error và redirect để frontend không bị vỡ luồng.
  • Không quên chain.doFilter(...): nếu không gọi, request sẽ dừng tại filter đó.

Kết luận

Filter trong dự án này là lớp cross-cutting đặt ở đầu pipeline request. Chúng không thay thế servlet/service, mà bổ sung các trách nhiệm chung như bảo mật, mã hóa, logging, CSRF, session restore, auth và RBAC. Khi đọc code, cứ đi theo thứ tự trong web.xml là sẽ thấy đúng luồng thực thi.
