🔒 Phân tích bảo mật

Điểm kiểm traKết quảChi tiết
Path Traversal trong upload AN TOÀN Tên file gốc bị bỏ qua hoàn toàn, UUID được dùng làm tên mới
MIME spoofing AN TOÀN Validate bằng extension, không tin Content-Type header
File size enforcement AN TOÀN Server-side check part.getSize() > 25MB + @MultipartConfig maxFileSize hai lớp
SQL Injection AN TOÀN Tất cả query dùng PreparedStatement với tham số, không nối chuỗi
XSS trong JSP AN TOÀN Dùng <c:out> cho mọi dữ liệu dynamic
CSRF trên multipart form AN TOÀN Kiểm tra CSRF token manual trong cả RegisterServlet và ShopApplyServlet
Duplicate submission (đăng ký 2 lần) AN TOÀN Kiểm tra findByUserId() trước khi tạo mới; PENDING → báo lỗi, APPROVED → báo lỗi, REJECTED → resubmit
Upload dir accessible trực tiếp CẦN CHÚ Ý uploads/shop-docs/ hiện accessible qua URL nếu Tomcat map static files. Khuyến nghị thêm servlet trung gian kiểm tra xác thực trước khi serve file.

⚖ Trade-offs & Giới hạn

  • 2 UPDATE riêng khi APPROVE: Do MSSQL không hỗ trợ UPDATE multi-table, dùng 2 Connection riêng thay vì 1 transaction atomically. Nếu bước 2 fail, bước 1 đã commit. Khuyến nghị bọc trong explicit transaction (Spring hoặc JDBC transaction manual) trong phiên bản tiếp theo.
  • userId=0 khi upload từ RegisterServlet: File được lưu vào uploads/shop-docs/0/ trước khi user được tạo. Nếu tạo user thất bại, file vẫn tồn tại trên server. Nên thêm cleanup job hoặc chuyển upload sau khi tạo user thành công.
  • preferred_categories lưu JSON string: Thay vì bảng join, lưu JSON đơn giản để tránh thêm bảng. Phù hợp vì dữ liệu này chỉ mang tính tham khảo cho Admin.
