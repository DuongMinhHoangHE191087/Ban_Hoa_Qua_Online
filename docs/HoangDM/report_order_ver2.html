▶ 1. Đặt Vấn Đề & Mục Tiêu Nâng Cấp

Trước bản cập nhật này, hệ thống gặp 3 bất cập nghiệp vụ nghiêm trọng:

  • Trạng thái bị treo: Sau khi khách thanh toán thành công (qua SePay webhook hoặc giả lập), đơn hàng vẫn hiển thị PENDING_PAYMENT — không phản ánh tiến độ thực tế cho khách.
  • Thiếu phân quyền rõ ràng: SePay webhook trước đây tự động đẩy thẳng sang CONFIRMED mà không qua bộ lọc nghiệp vụ — gây rủi ro khi có webhook giả mạo. Giải pháp mới: SePay được trao quyền ngang Admin đối với bước cụ thể này (xác nhận thanh toán VietQR đã khớp) nhưng được kiểm soát bởi logic xác minh nội tại tại PaymentService.java.
  • Giao diện quản trị thiếu bộ lọc: Admin không thể lọc nhanh đơn theo phương thức thanh toán hoặc trạng thái giao dịch, gây khó khăn đối soát dòng tiền.
✓ Mục tiêu đạt được trong bản cập nhật

Chuẩn hóa luồng 5 bước · Tách bạch Order Status & Payment Transaction Status · Tự động hóa SePay webhook có kiểm soát · Giao diện tracking trực quan · Hướng dẫn mở rộng bộ lọc cho mọi tầng DAO→Service→Servlet→JSP.

▶ 2. Phân Tích Luồng Nghiệp Vụ Chuẩn

A. Đơn hàng Chuyển khoản / QR (VietQR / Bank Transfer)

Luồng này yêu cầu xác minh giao dịch ngân hàng trước khi đơn được giao cho Shop để đóng gói.

# Bước hành trình Order Status Payment Status Hiển thị khách (Tracking) Tác nhân
1 Khởi tạo đơn PENDING_PAYMENT pending Chờ thanh toán (QR) Khách hàng
2a Khách báo đã chuyển PENDING_PAYMENT processing Đang chờ xác nhận ngân hàng Khách hàng
2b SePay khớp giao dịch CONFIRMED ← tự động completed ✓ Đã thanh toán — Chờ shop gửi hàng SePay Webhook
3 Shop đóng hàng PREPARING completed Shop đang chuẩn bị hàng Chủ Shop
4 Shipper lấy hàng DISPATCHED completed Đang giao hàng Shipper
5 Giao hàng thành công DELIVERED completed ✓ Hoàn thành Shipper / Hệ thống
💡 Tại sao SePay không cần Admin duyệt thêm?

SePay là cổng thanh toán xác thực ngân hàng trực tiếp — khi webhook gửi đến, giao dịch đã được ngân hàng xác nhận thành công (real-time notification). Việc bắt Admin duyệt thêm sẽ gây chậm trễ không cần thiết. Tuy nhiên, PaymentService.java vẫn xác minh nội tại (nội dung mô tả, số tiền, mã đơn hàng) trước khi cập nhật — đây là cơ chế phòng ngừa webhook giả mạo thay thế cho việc duyệt thủ công.

B. Đơn hàng COD (Thanh toán khi nhận hàng)

COD không có bước xác minh tài chính trước — đơn được xác nhận ngay và giao thẳng cho Shop.

1
Khởi tạo đơn COD Order Status: CONFIRMED  |  Khách thấy: "Đã xác nhận (COD), chờ shop gửi hàng"
2
Shop chuẩn bị & gửi hàng Order Status: PREPARING → DISPATCHED
3
Shipper giao + thu tiền mặt Order Status: DELIVERED  |  Payment Status: completed (thu COD)
⚠ Quyền Admin trong trường hợp webhook thất bại

Nếu SePay webhook không gọi được (mạng lỗi, timeout), Admin vẫn có thể thủ công chuyển đơn sang CONFIRMED qua màn hình quản trị đơn hàng — sau khi đối chiếu sao kê ngân hàng. Đây là fallback nghiệp vụ để không để đơn bị treo vô thời hạn.

▶ 4. Hướng Dẫn Tự Thêm Bộ Lọc (Developer Filter Guide)

Mẫu dưới đây dùng ví dụ lọc Sản phẩm theo khoảng giá (minPrice / maxPrice) tại trang Admin. Áp dụng tương tự cho bất kỳ thực thể nào trong hệ thống.

Thực hiện tuần tự 4 bước từ dưới lên trên (DAO → Service → Servlet → JSP):

1
Chỉnh sửa DAO — Thêm tham số lọc vào SQL động
Mở file DAO tương ứng, ví dụ: src/java/com/fruitmkt/dao/ProductDAO.java

Nguyên tắc: Dùng StringBuilder ghép SQL động + List<Object> params để tránh SQL Injection. Không bao giờ nối chuỗi trực tiếp vào SQL.

ProductDAO.java — phương thức findAll() với filter Java 
public List<Product> findAll(
        BigDecimal minPrice, BigDecimal maxPrice,
        int page, int pageSize) throws SQLException {

    List<Product> list = new ArrayList<>();
    int offset = (page - 1) * pageSize;

    // 1. SQL cơ bản — WHERE 1=1 cho phép ghép AND tùy ý
    StringBuilder sql = new StringBuilder(
        "SELECT * FROM products WHERE 1=1 ");
    List<Object> params = new ArrayList<>();

    // 2. Ghép điều kiện động theo tham số đầu vào
    if (minPrice != null) {
        sql.append("AND price >= ? ");
        params.add(minPrice);
    }
    if (maxPrice != null) {
        sql.append("AND price <= ? ");
        params.add(maxPrice);
    }

    // 3. Phân trang — SQL Server syntax
    sql.append("ORDER BY product_id DESC "
             + "OFFSET ? ROWS FETCH NEXT ? ROWS ONLY");
    params.add(offset);
    params.add(pageSize);

    // 4. Thực thi với try-with-resources (tránh connection leak)
    try (Connection conn = getConnection();
         PreparedStatement ps = conn.prepareStatement(sql.toString())) {

        for (int i = 0; i < params.size(); i++) {
            ps.setObject(i + 1, params.get(i));
        }
        try (ResultSet rs = ps.executeQuery()) {
            while (rs.next()) {
                list.add(mapRow(rs));
            }
        }
    }
    return list;
}

// countAll() — cùng logic WHERE, không cần OFFSET/FETCH
public int countAll(BigDecimal minPrice, BigDecimal maxPrice)
        throws SQLException {
    StringBuilder sql = new StringBuilder(
        "SELECT COUNT(*) FROM products WHERE 1=1 ");
    List<Object> params = new ArrayList<>();
    if (minPrice != null) { sql.append("AND price >= ? "); params.add(minPrice); }
    if (maxPrice != null) { sql.append("AND price <= ? "); params.add(maxPrice); }
    try (Connection conn = getConnection();
         PreparedStatement ps = conn.prepareStatement(sql.toString())) {
        for (int i = 0; i < params.size(); i++) ps.setObject(i + 1, params.get(i));
        try (ResultSet rs = ps.executeQuery()) {
            return rs.next() ? rs.getInt(1) : 0;
        }
    }
}

Đọc tham số tại Servlet — Parse & Validate an toàn
src/java/com/fruitmkt/servlet/admin/AdminProductServlet.java

Luôn dùng try-catch NumberFormatException khi parse số. Giá trị null là hợp lệ — có nghĩa là "không lọc theo trường này".

AdminProductServlet.java — doGet() Java 
@Override
protected void doGet(HttpServletRequest req, HttpServletResponse resp)
        throws ServletException, IOException {

    // --- 1. Đọc tham số filter từ query string ---
    String minPriceStr = req.getParameter("minPrice");
    String maxPriceStr = req.getParameter("maxPrice");
    String pageStr     = req.getParameter("page");

    // --- 2. Parse an toàn (null = không lọc) ---
    BigDecimal minPrice = null, maxPrice = null;
    int page = 1, pageSize = 20;

    try {
        if (minPriceStr != null && !minPriceStr.isBlank())
            minPrice = new BigDecimal(minPriceStr.trim());
        if (maxPriceStr != null && !maxPriceStr.isBlank())
            maxPrice = new BigDecimal(maxPriceStr.trim());
        if (pageStr != null)
            page = Math.max(1, Integer.parseInt(pageStr));
    } catch (NumberFormatException e) {
        // Bỏ qua giá trị không hợp lệ, dùng default
    }

    // --- 3. Gọi service & đẩy sang JSP ---
    try {
        List<Product> products =
            productService.getProducts(minPrice, maxPrice, page, pageSize);
        int total = productService.countProducts(minPrice, maxPrice);

        req.setAttribute("products", products);
        req.setAttribute("totalPages", (total + pageSize - 1) / pageSize);
        req.setAttribute("currentPage", page);
        req.setAttribute("minPrice", minPrice);
        req.setAttribute("maxPrice", maxPrice);

        req.getRequestDispatcher("/WEB-INF/jsp/admin/products.jsp")
           .forward(req, resp);
    } catch (SQLException e) {
        throw new ServletException("Lỗi truy vấn sản phẩm", e);
    }
}

💡 Lưu ý: tham số phải truyền qua link phân trang

Khi người dùng chuyển trang, URL phải giữ nguyên các tham số filter bằng cách append &amp;minPrice=${minPrice}&amp;maxPrice=${maxPrice} vào mỗi link trang. Nếu không, trang tiếp theo sẽ mất bộ lọc.
