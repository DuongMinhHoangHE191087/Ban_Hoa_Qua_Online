2. Kiến Trúc Xác Thực Dual-Token

Hệ thống xác thực mới hoạt động trên nguyên tắc kết hợp giữa Access Token ký số tự chứa (self-contained signed token) và Refresh Token lưu trữ đối chiếu tại Database.

A. Access Token ký số (HMAC-SHA256)

Được thiết kế gọn nhẹ, không cần thư viện ngoài giúp dự án biên dịch siêu tốc. Token có định dạng:

userId.expiresAt.signature

Trong đó signature là mã băm băm ngược userId.expiresAt bằng thuật toán HmacSHA256 với khóa bí mật cực kỳ an toàn tại TokenUtil.java. Cookie được thiết lập thuộc tính HttpOnly để ngăn chặn triệt để các cuộc tấn công đánh cắp token bằng XSS.

B. Refresh Token & Session Database

Có hiệu lực dài hạn (7 ngày), được biểu diễn dưới dạng chuỗi ngẫu nhiên bảo mật cao (UUID) và lưu trữ đối chiếu trong bảng user_sessions:

// Định nghĩa các phương thức tại UserDAO.java
public void saveUserSession(int userId, String token, Timestamp expiresAt) throws SQLException;
public Integer findUserIdBySessionToken(String token) throws SQLException;
public void deleteUserSession(String token) throws SQLException;

3. Cơ Chế Tự Động Làm Mới Phiên Chạy (Silent Refresh Flow)

Bảo vệ toàn bộ các khu vực nhạy cảm (/customer/*, /shop/*, /delivery/*, /admin/*) thông qua AuthFilter.java:

1
Yêu cầu truy cập: Người dùng truy cập Protected URL (Ví dụ: /shop/dashboard).
2
Kiểm tra session: Nếu HTTP Session còn hiệu lực, cho qua luôn (Hiệu năng tối ưu 100%).
3
Kiểm tra Access Token: Nếu Session hết hạn nhưng accessToken cookie còn hiệu lực -> Xác thực chữ ký Hmac -> Tự động khôi phục Session mà không cần truy vấn Database!
4
Silent Refresh: Nếu accessToken hết hạn nhưng refreshToken cookie tồn tại -> Đối chiếu DB -> Nếu hợp lệ, sinh và ghi cookie accessToken mới -> Tự động khôi phục session!
5
Chuyển hướng an toàn: Nếu cả hai token đều hết hạn/không hợp lệ -> Điều hướng sang /auth/login?redirect=[URL_gốc_đã_mã_hóa].

4. Lá Chắn Bảo Mật Tích Hợp

  • Chống Tấn Công Session Fixation: Khi đăng nhập thành công tại LoginServlet.java, hệ thống tự động gọi session.invalidate() trước khi cấp Session ID mới.
  • Xác Thực CSRF Chặt Chẽ: Xác thực mã bảo mật _csrf ẩn từ form đăng nhập chống giả mạo request chéo trang.
  • Ngăn Chặn Timing Attack: Sử dụng phương thức so sánh an toàn thời gian thực MessageDigest.isEqual để đối chiếu chữ ký Access Token trong TokenUtil.java.
  • Chặn Lỗ Hổng Open Redirect: Kiểm tra tính hợp lệ của tham số redirect (chỉ cho phép các đường dẫn nội bộ bắt đầu bằng / hoặc Context Path), chặn đứng hacker điều hướng nạn nhân sang web lừa đảo bên ngoài.

5. Quy Trình Kiểm Thử Xác Minh (E2E Manual Testing)

  1. Kiểm tra Login & Cookies: Mở trình duyệt ẩn danh, đăng nhập tại /auth/login. F12 -> Application -> Cookies. Đảm bảo xuất hiện accessToken và refreshToken có cờ HttpOnly.
  2. Kiểm tra Phân Quyền Redirection:
    • Tài khoản ADMIN tự động chuyển về /admin/dashboard.
    • Tài khoản SHOP_OWNER tự động chuyển về /shop/dashboard.
    • Tài khoản CUSTOMER tự động chuyển về trang chủ /.
  3. Kiểm tra Silent Refresh: Đăng nhập thành công, xóa cookie accessToken (giả lập hết hạn 15 phút), nhấn F5 tải lại trang Dashboard. Bạn sẽ thấy trang tải bình thường và accessToken tự động được sinh mới lại trong Cookies mà bạn không hề phải đăng nhập lại!
  4. Kiểm tra Logout: Click Đăng xuất. Đảm bảo bản ghi Refresh Token bị xóa sạch khỏi bảng user_sessions ở Database, cookies bị xóa, và bạn được chuyển hướng về trang Login kèm thông báo Flash thành công tuyệt đẹp!
