Tài liệu thực hành để lần theo code, tạo API đúng chuẩn và bảo vệ luồng mua hàng. Nguyên tắc quan trọng: không “né” CSRF bằng cách bypass toàn bộ /api/*; phải phân biệt request dùng cookie session, request dùng bearer token và webhook từ bên thứ ba.

3. CSRF: cách xử lý đúng và cách gọi API

Trong project này, session được giữ bằng cookie và filter tạo token tại src/java/filter/CsrfFilter.java. Vì vậy API của browser vẫn cần CSRF. Token có thể gửi bằng hidden field hoặc header; header phù hợp hơn cho fetch.

3.1. Server-side validation chuẩn

Đoạn dưới đây là mẫu đã áp dụng ở filter trung tâm: token được so sánh constant-time và dùng UTF-8 rõ ràng.

String sessionToken = (String) session.getAttribute(AppConfig.SESSION_CSRF_TOKEN);
String requestToken = req.getParameter("_csrf");
if (requestToken == null || requestToken.trim().isEmpty()) {
    requestToken = req.getHeader("X-CSRF-Token");
}

boolean validToken = sessionToken != null
        && requestToken != null
        && MessageDigest.isEqual(
                sessionToken.getBytes(StandardCharsets.UTF_8),
                requestToken.getBytes(StandardCharsets.UTF_8));
if (!validToken) {
    resp.setStatus(HttpServletResponse.SC_FORBIDDEN);
    JsonUtil.writeJson(resp, ApiResponse.fail(403,
            "CSRF token không hợp lệ.", Map.of("errorCode", "csrf_invalid")));
    return;
}

3.2. Client gọi API không bị 403

const csrf = document.querySelector('input[name="_csrf"]')?.value;
const response = await fetch(contextPath + '/api/checkout/quote', {
  method: 'POST',
  credentials: 'same-origin',
  headers: {
    'Accept': 'application/json',
    'Content-Type': 'application/json',
    'X-CSRF-Token': csrf
  },
  body: JSON.stringify({
    cartItemIds: selectedCartItemIds,
    deliveryAddress,
    paymentMethod,
    shopCouponCodes,
    systemCouponCodes
  })
});
const body = await response.json();
if (!response.ok || !body.success) {
  // response.status quyết định nhánh kỹ thuật; body.meta.errorCode quyết định UX.
  showError(body.meta?.errorCode || 'request_failed');
}
Không “tránh CSRF” bằng bypass: chỉ webhook bên ngoài được miễn vì không nhận cookie session. Nếu sau này dùng access token trong Authorization: Bearer và không dùng cookie, CSRF có thể không áp dụng cho route đó; khi ấy phải có expiry, revoke, scope và không để token trong URL.

4. Response contract để frontend và hội đồng dễ kiểm tra

Project đã có src/java/model/response/ApiResponse.java với dạng success, data, error, meta. Giữ dạng này để không phá client, nhưng HTTP status phải được set thật trên response.

// Thành công
HTTP/1.1 200 OK
{"success":true,"data":{"quote":{}},"meta":null}

// Lỗi validation
HTTP/1.1 400 Bad Request
{"success":false,"error":"Dữ liệu không hợp lệ.",
 "meta":{"statusCode":400,"errorCode":"invalid_request"}}

// Không còn stock / xung đột trạng thái
HTTP/1.1 409 Conflict
{"success":false,"error":"Sản phẩm vừa hết hàng.",
 "meta":{"statusCode":409,"errorCode":"out_of_stock"}}

// Vi phạm business rule
HTTP/1.1 422 Unprocessable Entity
{"success":false,"error":"Không thể đổi trả ngoài thời hạn.",
 "meta":{"statusCode":422,"errorCode":"return_window_expired"}}

Code mẫu cho servlet:

resp.setStatus(HttpServletResponse.SC_CONFLICT);
JsonUtil.writeJson(resp, ApiResponse.fail(
        HttpServletResponse.SC_CONFLICT,
        "Một số sản phẩm không còn đủ tồn kho.",
        Map.of("errorCode", "out_of_stock")));
return;
Gap đã sửa: CartServlet#doPost không còn trả lỗi business bằng HTTP 200; stock conflict trả 409, input trả 400, validation trả 422 và exception hệ thống trả 500. Frontend hiện vẫn đọc body.success, nên thay đổi này không làm mất UX cũ.

5.2. Khung servlet có thể dùng để triển khai

Phân biệt code mẫu và code production: ReturnRequestMapper trong đoạn dưới là placeholder để chỉ rõ seam parse/validate; class này chưa tồn tại trong source hiện tại. Phải hoàn thiện evidence storage và test service trước khi tạo servlet thật.
@WebServlet("/api/v1/returns")
@MultipartConfig(maxFileSize = 50L * 1024 * 1024,
                 maxRequestSize = 150L * 1024 * 1024)
public class ReturnRequestApiServlet extends HttpServlet {
    private final ReturnService returnService = new ReturnService();

    @Override
    protected void doPost(HttpServletRequest req, HttpServletResponse resp)
            throws IOException, ServletException {
        resp.setContentType("application/json;charset=UTF-8");
        User user = SessionUtil.getCurrentUser(req.getSession(false));
        if (user == null) {
            resp.setStatus(HttpServletResponse.SC_UNAUTHORIZED);
            JsonUtil.writeJson(resp, ApiResponse.fail(401, "Vui lòng đăng nhập.",
                    Map.of("errorCode", "unauthenticated")));
            return;
        }
        if (!ActorAccessPolicy.canCreateReturnRequest(user)) {
            resp.setStatus(HttpServletResponse.SC_FORBIDDEN);
            JsonUtil.writeJson(resp, ApiResponse.fail(403, "Không có quyền đổi trả.",
                    Map.of("errorCode", "forbidden")));
            return;
        }

        // CSRF do CsrfFilter kiểm tra trước; không tạo bypass trong servlet.
        // Parse + validate multipart, sau đó gọi cùng ReturnService như form PRG.
        // Evidence phải được lưu an toàn trước khi commit DB; không lưu filename thô.
        ReturnRequest request = ReturnRequestMapper.fromMultipart(req, user);
        int id = returnService.createRequest(request);

        resp.setStatus(HttpServletResponse.SC_CREATED);
        JsonUtil.writeJson(resp, ApiResponse.ok(
                Map.of("returnRequestId", id, "status", "REQUESTED")));
    }
}
Không copy nguyên mẫu và tuyên bố hoàn tất: hiện schema chỉ có một evidence_url, trong khi rule yêu cầu 1 video hoặc 2 ảnh. Muốn API đạt chuẩn phải có bảng return_request_evidence hoặc JSON schema được validate, lưu file an toàn và cleanup file nếu transaction DB rollback.

6. Transaction và idempotency: phần hội đồng thường hỏi

6.1. Vì sao code hiện tại chưa đủ

ReturnService#createRequest/decide và ReturnRequestService#processRequest gọi các DAO mở connection riêng. Nếu cập nhật request thành công nhưng cập nhật refund status hoặc inventory thất bại, DB có thể lệch trạng thái.

6.2. Mẫu DAO bắt buộc nhận Connection

// DAO: SQL vẫn ở DAO, nhưng dùng connection do Service quản lý
public void updateStatus(Connection conn, int id, String status,
                         String reason, int adminId) throws SQLException {
    String sql = "UPDATE return_requests SET status = ?, decision_reason = ?, "
               + "decided_by = ?, updated_at = GETDATE() "
               + "WHERE return_request_id = ?";
    try (PreparedStatement ps = conn.prepareStatement(sql)) {
        ps.setString(1, status);
        ps.setString(2, reason);
        ps.setInt(3, adminId);
        ps.setInt(4, id);
        ps.executeUpdate();
    }
}

6.3. Mẫu Service transaction

try (Connection conn = returnRequestDAO.getConnection()) {
    conn.setAutoCommit(false);
    try {
        ReturnRequest rr = returnRequestDAO.findById(conn, requestId);
        // Kiểm tra status transition + ownership từ DB, không từ form.
        returnRequestDAO.updateStatus(conn, requestId, "PROCESSING", reason, adminId);
        orderDAO.updateRefundStatus(conn, rr.getOrderId(), "PENDING");
        // Inventory/refund ledger cùng connection nếu nghiệp vụ yêu cầu.
        conn.commit();
    } catch (Exception e) {
        conn.rollback();
        throw e;
    } finally {
        conn.setAutoCommit(true);
    }
}

Thanh toán hiện tại đã có dedup theo transaction SePay và đối chiếu amount tuyệt đối. Đối với refund, cần thêm idempotency key hoặc unique refund transaction để double-click complete không hoàn tiền hai lần.

8. Checklist kiểm thử và lệnh tìm code

8.1. Test bắt buộc

  • ☐ POST cart/quote/checkout/review/return thiếu CSRF: 403 JSON hoặc 403 page đúng loại request.
  • ☐ CSRF gửi bằng hidden field và X-CSRF-Token đều pass; token sai/khác session fail.
  • ☐ Customer A dùng ID của customer B không đọc/sửa được cart, order, review, return.
  • ☐ API malformed JSON/content type quá lớn/unknown field bị reject theo contract.
  • ☐ Stock race trả 409; invalid business trả 422; client vẫn hiển thị đúng meta.errorCode.
  • ☐ Admin complete không nhận orderId giả; orderId được load từ return request trong DB.
  • ☐ Complete/retry đồng thời không tạo refund hai lần; rollback không để request và refund status lệch.

8.3. Những file đã sửa trong lượt nâng cấp này

  • src/java/filter/CsrfFilter.java: constant-time compare bằng UTF-8.
  • src/java/servlet/api/cart/CheckoutQuoteServlet.java và src/java/servlet/customer/cart/CheckoutServlet.java: compare token với charset rõ ràng.
  • src/java/servlet/customer/cart/CartServlet.java: status lỗi 400/409/422/500 thay cho 200.
  • src/java/service/order/ReturnRequestService.java và src/java/servlet/admin/order/AdminRefundServlet.java: derive orderId từ requestId, không tin hidden input; reject action/reason không hợp lệ.

Source of truth: filter/web.xml cho boundary; Servlet cho HTTP contract; Service cho business rule/transaction; DAO/schema cho persistence constraint. Không dùng tài liệu cũ thay cho code hiện tại nếu hai bên khác nhau.
