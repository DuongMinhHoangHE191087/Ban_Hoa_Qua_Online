Kiểm tra luồng hệ thống, công nghệ và điểm nâng cấp

Tài liệu này gom lại các luồng chính đang có trong repo: đăng ký, xác minh email, đăng nhập, JavaMail, Java Activation, SePay xác nhận giao dịch, tải sản phẩm và AI hỗ trợ tìm kiếm. Mục tiêu là đọc theo luồng request thật, không đọc theo cảm tính.

→

2. Luồng đăng ký và kích hoạt email

Luồng này kết nối RegisterServlet, AuthService, UserDAO, EmailService và VerifyEmailServlet. Tài khoản được tạo trước ở trạng thái chưa kích hoạt, sau đó mới mở khóa bằng mã email.

Submit form đăng ký

POST /auth/register. Servlet kiểm tra CSRF, sanitize input và validate các trường bắt buộc.

→

Tạo user inactive

AuthService.register() kiểm tra trùng email, trùng số điện thoại, hash mật khẩu rồi lưu user với trạng thái chưa kích hoạt.

→

Phát mã xác minh

issueVerificationCode() lưu hash mã OTP cùng expiry/resend timestamp rồi gọi EmailService.

→

Xác minh email

GET/POST /auth/verify cho phép resend hoặc verify. Mã đúng thì account chuyển sang active.

→

Kích hoạt hoàn tất

UserDAO.activateVerifiedEmail() set status = ACTIVE và is_email_verified = 1.

Điểm mạnh hiện tại

  • Validate và sanitize trước khi persistence.
  • Mật khẩu được hash trước khi đưa xuống DAO.
  • Luồng verify dùng hash của mã xác minh, không lưu mã thô.
  • Shop owner có nhánh lưu pendingShopProfile trong session trước khi kích hoạt xong.

Điểm cần đọc kỹ

  • Email verification là gate bắt buộc trước login.
  • Khách đăng ký shop sẽ upload tài liệu và chờ profile được lưu sau khi verify.
  • Verify thành công có thể kích hoạt thêm gửi mail nhận đơn shop theo thread nền.
Giải thích về "activation" Trong code hiện tại, activation không phải là một package hay class riêng. Nó là trạng thái nghiệp vụ: email đã xác minh, account chuyển sang ACTIVE, và cờ is_email_verified được bật. Nói ngắn gọn, "kích hoạt" là flow business, còn Java Activation chỉ là dependency nền cho mail stack.

3. Luồng đăng nhập và bảo vệ session

LoginServlet làm khá đúng vai trò controller mỏng: nhận input, kiểm CSRF, gọi service, rồi quyết định redirect. Phần logic an toàn chính nằm ở AuthService.login().

Nhận credentials

POST /auth/login nhận identifier và password. Số điện thoại được normalize nếu người dùng nhập theo dạng 0, +84 hoặc 84.

→

Chống CSRF

Servlet so sánh token session với token từ form trước khi gọi service.

→

Kiểm tra lock và hash

AuthService.login() chặn tài khoản đang bị lock, so hash mật khẩu, tăng failed count nếu sai.

→

Chưa verify thì chặn

Nếu account chưa active hoặc chưa xác minh email, service ném VerificationRequiredException.

→

Đăng nhập thành công

Invalidate session cũ, tạo session mới, cấp access token + refresh token và redirect theo role.

Điểm an toàn đã có

  • Session fixation defense bằng cách hủy session cũ.
  • Open redirect bị chặn bằng kiểm tra target path.
  • Lock account khi nhập sai quá nhiều lần.

Điểm cần hiểu đúng

  • Đăng nhập không chỉ kiểm mật khẩu, còn kiểm trạng thái verify.
  • Role quyết định redirect cuối cùng: admin, shop owner, delivery, customer.
  • Refresh token được lưu DB thông qua service session logic.

Lỗi người dùng thường gặp

  • Nhập đúng mật khẩu nhưng chưa xác minh email.
  • Quá nhiều lần sai mật khẩu nên tài khoản bị khóa tạm thời.
  • Identifier nhập là số điện thoại nhưng chưa đúng format chuẩn hóa.

4. JavaMail và Java Activation

Tầng email hiện tại tập trung vào gửi HTML mail qua SMTP. JavaMail là phần trực tiếp trong code; Java Activation là dependency nền của stack mail, hữu ích nhất khi cần MIME, attachment hoặc inline content.

Session Tạo phiên SMTP với auth, STARTTLS, timeout và trust host.
MimeMessage Đóng gói tiêu đề, recipient, nội dung HTML và định dạng UTF-8.
Transport Thực sự gửi mail ra SMTP server, sau đó đóng kết nối an toàn.
Template EmailTemplateService build nội dung, EmailService chỉ lo transport.

Các kiểu mail đang dùng

  • Mã xác minh sau đăng ký.
  • Mã quên mật khẩu.
  • Thông báo đơn hàng cho customer và shop owner.
  • Email nhận đơn đăng ký shop và trạng thái duyệt/từ chối.

Java Activation nên hiểu thế nào

Repo hiện không có luồng nghiệp vụ riêng cho Activation. Nó là lớp hỗ trợ MIME/attachment của mail stack. Nếu tương lai thêm file đính kèm, ảnh inline hoặc mail dạng đa phần, phần này sẽ thành rõ vai trò hơn.

Rủi ro vận hành Gửi mail đang nằm ngay trong request path ở nhiều chỗ. Nếu SMTP chậm hoặc lỗi, response có thể chậm theo. Về lâu dài nên đưa sang hàng đợi hoặc outbox để tách độ trễ mail khỏi luồng chính.

5. Luồng checkout và SePay xác nhận giao dịch

Đây là luồng quan trọng nhất về tiền. Hệ thống đã có chốt chống trùng webhook, kiểm tra số tiền và chỉ chuyển đơn sang confirmed khi thanh toán khớp điều kiện.

Checkout tạo order

CheckoutService.placeOrder() tạo đơn. Nếu payment method là COD thì đơn vào trạng thái confirmed ngay, còn CK thì vào PENDING_PAYMENT.

→

Tạo payment transaction

PaymentService.initPayment() sinh reference SePay dạng MFxxx, lưu payment_transactions và expiry.

→

Khách chuyển khoản

QR hiển thị reference đó để SePay map ngược giao dịch về order tương ứng.

→

SePay gọi webhook

POST /api/payment/webhook nhận JSON payload và luôn phản hồi HTTP 200 để tránh retry do gateway.

→

Đối soát và xác nhận

PaymentService.processWebhook() dedup theo transaction id, kiểm amount, kiểm status, rồi cập nhật order sang CONFIRMED.

Điểm đang làm đúng

  • Dedup webhook bằng transaction id của SePay.
  • Chỉ xử lý chuyển tiền vào, bỏ qua các chiều khác.
  • Chỉ xác nhận nếu order vẫn đang chờ thanh toán.
  • Kiểm tra số tiền thực nhận không được nhỏ hơn số tiền mong đợi.

Ý nghĩa của trạng thái

  • PENDING_PAYMENT - đơn chờ webhook xác nhận.
  • completed - payment transaction đã khớp.
  • CONFIRMED - đơn đã được xác nhận thanh toán.
  • COD bỏ qua webhook vì không cần chờ cổng thanh toán.
Điểm cần theo dõi chặt Webhook hiện trả 200 ngay cả khi xử lý nội bộ lỗi. Cách này giúp tránh SePay retry, nhưng đồng thời làm lỗi chỉ còn xuất hiện trong log nội bộ. Nếu muốn vận hành bền hơn, cần alert riêng cho payload lỗi.

6. Luồng tải, thêm và chỉnh sửa sản phẩm

Phần này thuộc shop owner. Hiện có luồng tạo mới, sửa đổi, upload ảnh, upload giấy tờ xác minh và luồng admin phê duyệt hoặc từ chối. Đây là nơi cần giữ transaction và cleanup file cẩn thận.

Shop owner mở form

/shop/product-create hoặc /shop/product-edit. MultipartConfig giới hạn kích thước request và file.

→

Validate dữ liệu

Tên, danh mục, biến thể, giá, ảnh, giấy tờ xác minh và các trường mùa vụ được kiểm tra trước khi lưu.

→

Upload file

FileUploadUtil lưu ảnh và doc xuống thư mục webapp, sau đó DAO lưu path vào DB.

→

Ghi DB

Sản phẩm, ảnh, biến thể và bao bì tùy chọn được lưu qua DAO. Sản phẩm mới hoặc sửa đổi đều có trạng thái kiểm duyệt.

→

Admin duyệt

AdminProductServlet cùng ProductService duyệt, từ chối hoặc gỡ bỏ sản phẩm vi phạm.

Luồng tạo mới

  • Bắt buộc có tối thiểu một biến thể.
  • Ảnh và giấy tờ xác minh được upload cùng request.
  • Sản phẩm mới được tạo ở trạng thái ACTIVE nhưng chờ duyệt ở PENDING.

Luồng chỉnh sửa

  • Chỉ owner của sản phẩm được sửa.
  • Xóa biến thể trên UI sẽ bị deactive.
  • Bất kỳ thay đổi nào cũng đưa approval về PENDING.

Luồng quản trị

  • approve - duyệt và đưa lên sàn.
  • reject - lưu lý do từ chối.
  • ban - gỡ bỏ sản phẩm vi phạm.
Rủi ro cần lưu ý Upload file đang đi chung với lưu DB nên nếu giữa chừng lỗi, rất dễ sinh file rác trên disk. Nếu hệ thống tăng tải, nên tách file storage ra object storage hoặc có job dọn file orphan.

7. AI hỗ trợ tìm sản phẩm

AI ở đây là trợ lý tìm kiếm - tư vấn sản phẩm, không phải nguồn dữ liệu gốc. Dữ liệu chuẩn vẫn là DB; AI chỉ đọc catalog đang active và trả về reply cùng danh sách ID gợi ý.

Lấy API key

AiSearchServlet tìm key từ DB, biến môi trường hoặc file .env.

→

Nạp catalog

Đọc danh mục active và danh sách sản phẩm brief đang còn bán để làm context cho prompt.

→

Gọi Gemini

Xây prompt hệ thống, bắt schema JSON với reply và suggestedProductIds.

→

Retry khi lỗi mạng

Tự động retry nếu gặp HTTP 503 hoặc 429 trước khi trả lỗi cho frontend.

→

Lọc lại kết quả

Chỉ giữ ID của sản phẩm thực sự active, giới hạn số gợi ý hiển thị ở mức vừa phải.

Điểm đáng ghi nhận

  • Prompt có schema đầu ra rõ ràng.
  • Kết quả AI không được phép bịa ID không tồn tại.
  • Frontend nhận luôn cả reply và danh sách product để render tốt hơn.

Cần nhớ khi debug

  • Thiếu key thì AI fail ngay từ đầu.
  • API upstream chậm thì retry mới giúp giảm lỗi ngẫu nhiên.
  • Dữ liệu AI chỉ là gợi ý, không phải nguồn quyết định cuối cùng.

8. Đánh giá hiện trạng

Bảng dưới là nhìn nhanh theo góc độ vận hành, không phải chấm điểm học thuật.

Luồng Đã có gì Điểm cần theo dõi
Đăng ký + xác minh email Tạo user inactive, gửi mã xác minh, kích hoạt bằng code hash, có nhánh shop owner. Resend rate limit, cleanup nếu upload file/shop profile lỗi giữa chừng.
Đăng nhập CSRF, lock account, chặn chưa verify, session fixation defense, token hóa login. Chưa thấy MFA và chưa thấy dashboard cảnh báo brute force.

SePay Init payment, dedup webhook, amount check, order confirmed tự động. Chưa thấy lớp signature validation rõ trong phần code đã đọc.

Kết luận thực dụng: hệ thống đã đi đúng hướng nghiệp vụ, nhưng phần bền vận hành còn dư địa lớn để nâng cấp.

9. Gợi ý nâng cấp ưu tiên

Các đề xuất dưới đây được xếp theo mức ảnh hưởng thực tế và mức độ đáng làm trước.

P1 - Tách email ra hàng đợi

  • Đưa gửi mail sang queue/outbox để không block request.
  • Thêm retry có giới hạn và log correlation id.
  • Dễ quan sát hơn khi SMTP lỗi hoặc bị throttle.

P1 - Cứng hóa webhook SePay

  • Thêm signature validation hoặc whitelist nguồn nếu SePay hỗ trợ.
  • Bắn alert khi webhook fail nội bộ dù HTTP vẫn trả 200.
  • Đảm bảo unique index cho transaction id và reference.

P2 - Dọn dẹp upload file

  • Tách storage sang object storage hoặc media service.
  • Có job dọn file orphan sau rollback hoặc submit lỗi.
  • Bổ sung scan file an toàn hơn nếu cho phép upload tài liệu.

P2 - Theo dõi luồng auth

  • Thêm audit trail cho đăng ký, resend code, login fail và lock/unlock.
  • Có dashboard nhìn số lần verify fail, login fail, resend fail.
  • Dễ trả lời khi support user hỏi "vì sao không vào được".

P3 - Tăng độ tin cậy AI

  • Cache câu trả lời cho các truy vấn lặp lại.
  • Gắn timeout và fallback text rõ ràng khi upstream chậm.
  • Log câu hỏi, catalog snapshot và output để tối ưu prompt.

P3 - Bổ sung regression tests

  • Auth register/verify/login/lock.
  • SePay webhook dedup, amount mismatch, wrong status.
  • Product upload, edit, approve, reject, ban.
Khuyến nghị thực tế nhất Nếu chỉ làm 3 việc trước, hãy chọn: 1) email queue, 2) SePay alerting/signature hardening, 3) cleanup file upload. Đây là 3 điểm giảm rủi ro cao nhất cho hệ thống hiện tại.

10. Checklist kiểm tra nhanh

Dùng checklist này khi muốn tự test từng luồng mà không phải đọc toàn bộ code.

Auth registration Kiểm tra CSRF, validate email/phone, lưu user inactive, gửi mã xác minh và chuyển sang trang verify.
Email verification Thử resend, thử code sai, thử code hết hạn, thử code đúng và xác nhận user active.
Login Thử đăng nhập sai nhiều lần để xem lock, thử tài khoản chưa verify, thử redirect theo role.
SePay Test pending payment, dedup webhook, wrong amount, wrong status và confirm chuyển sang CONFIRMED.
Product upload Upload ảnh hợp lệ, ảnh sai định dạng, thiếu giấy tờ xác minh, update sản phẩm và kiểm tra approval reset.
AI Test mất key, 429/503, reply hợp lệ, suggested IDs có thực và giới hạn số gợi ý.
