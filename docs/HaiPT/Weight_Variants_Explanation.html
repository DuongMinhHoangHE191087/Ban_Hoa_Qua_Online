Tổng quan chức năng & Luồng dữ liệu (Data Flow)

Feature: Weight Variants Management
Giải thích cách hệ thống kết hợp Entity, DAO, Servlets và JSPs để tạo nên trải nghiệm quản lý phân loại sản phẩm.

🔄 Quy trình lưu trữ và hiển thị biến thể

Mỗi sản phẩm hoa quả sạch có thể bán theo các quy cách đóng gói (cân nặng) khác nhau. Thay vì lưu chung một giá và một tồn kho cứng ở bảng sản phẩm, hệ thống tách riêng ra bảng product_variants. Mỗi bản ghi variant sẽ gắn với 1 productId và tự chủ động lưu giữ SKU, nhãn mô tả (VD: "500g", "Hộp 1kg"), đơn giá gốc, tồn kho thực tế, cân nặng quy đổi (kg) và các cấu hình khuyến mại.

📦 Các thuộc tính ánh xạ DB
  • variantId: Khóa chính tự tăng, định danh duy nhất của mỗi variant trong DB.
  • productId: Khóa ngoại trỏ đến bảng products, liên kết variant với sản phẩm mẹ.
  • sku: Mã định danh sản phẩm duy nhất (Stock Keeping Unit), dùng cho quản lý logistics.
  • variantLabel: Nhãn hiển thị quy cách đóng gói như: "Hộp 500g", "Túi 1kg", "Thùng 5kg".
  • price: Giá bán gốc của biến thể (kiểu BigDecimal để chống sai số dấu phẩy động).
  • stockQuantity: Số lượng tồn kho hiện tại của riêng biến thể này.
  • isActive: Cờ boolean đánh dấu hoạt động, dùng để xóa mềm (soft delete) variant.
  • weightKg: Cân nặng thực tế quy đổi ra kg, dùng để tính phí vận chuyển theo trọng lượng đơn hàng.
  • discountPrice, discountStart, discountEnd: Giá khuyến mãi và thời gian áp dụng ưu đãi của biến thể.

💾 Giải thích chi tiết hàm save()
  • Dòng 126: Viết câu lệnh INSERT tĩnh, truyền các giá trị qua ký tự hỏi chấm ? để ngăn chặn triệt để SQL Injection.
  • Dòng 128: Sử dụng Statement.RETURN_GENERATED_KEYS để yêu cầu SQL Server trả lại giá trị ID tự sinh (cột variant_id) sau khi chèn thành công.
  • Dòng 134: Xử lý an toàn: nếu cân nặng weightKg truyền vào bị null thì gán giá trị mặc định là 1.0 kg.
  • Dòng 137-138: Chuyển đổi định dạng ngày của Java 8 LocalDateTime sang định dạng database java.sql.Timestamp. Kiểm tra điều kiện null để tránh lỗi NullPointerException.
  • Dòng 140-144: Đọc luồng ResultSet trả về từ getGeneratedKeys() để lấy ra khóa chính tự tăng của biến thể vừa thêm và trả về giá trị ID này cho Servlet sử dụng tiếp.
decrementStockQuantity() — Trừ kho nguyên tử UPDATE WITH OUTPUT
235 public int decrementStockQuantity(Connection conn, int variantId, int qty) throws SQLException {
236 String sql = "UPDATE product_variants SET stock_quantity = stock_quantity - ?, updated_at = GETDATE() "
237 + "OUTPUT INSERTED.stock_quantity "
238 + "WHERE variant_id = ? AND stock_quantity >= ? AND is_active = 1";
239 try (PreparedStatement ps = conn.prepareStatement(sql)) {
240 ps.setInt(1, qty);
241 ps.setInt(2, variantId);
242 ps.setInt(3, qty);
243 try (ResultSet rs = ps.executeQuery()) {
244 if (rs.next()) {
245 return rs.getInt("stock_quantity");
246 }
247 }
248 }
249 throw new SQLException("Không đủ số lượng hàng tồn kho cho sản phẩm hoặc sản phẩm không hoạt động.");
250 }
⚠️ Kỹ thuật chống bán quá lượng tồn kho (Race Condition)

Đây là câu lệnh SQL cực kỳ quan trọng để đảm bảo tính nhất quán dữ liệu khi có nhiều khách hàng mua cùng 1 sản phẩm tại 1 thời điểm:

  • Dòng 235: Nhận vào Connection conn dùng chung cho toàn bộ transaction thanh toán (được đóng/mở ở lớp Service).
  • Dòng 236-238: Kết hợp điều kiện stock_quantity >= ? trực tiếp ở mệnh đề WHERE. Điều này đảm bảo việc kiểm tra số lượng và trừ kho là một thao tác nguyên tử (atomic operation) ở mức database. Nếu tồn kho hiện tại nhỏ hơn số lượng khách mua qty, câu lệnh UPDATE sẽ tác động đến 0 dòng.
  • Mệnh đề OUTPUT INSERTED.stock_quantity cho phép cập nhật dữ liệu đồng thời trả về giá trị số lượng tồn kho mới sau khi trừ ngay trong cùng một request DB, tối ưu hóa hiệu năng hệ thống.
  • Dòng 249: Nếu rs.next() trả về rỗng (tức UPDATE không được thực hiện do không khớp điều kiện WHERE), hệ thống lập tức ném lỗi ngoại lệ để kích hoạt rollback toàn bộ transaction thanh toán đơn hàng.

🔄 Kỹ thuật đối soát đồng bộ danh sách

Khi chủ shop bấm cập nhật thông tin sản phẩm, họ có thể sửa đổi biến thể cũ, thêm biến thể mới hoặc xóa đi một vài dòng biến thể hiện có trên giao diện. Xử lý đồng bộ được chia làm 2 giai đoạn:

Giai đoạn 1: Đọc toàn bộ các variantId được gửi lên từ form vào tập hợp submittedIds. Lấy danh sách các biến thể đang lưu ở DB ra đối chiếu. Nếu ID nào ở DB không có trong danh sách form gửi lên, tức là chủ shop đã nhấn nút Xóa dòng đó trên UI. Servlet sẽ gọi deactivate() để thực hiện cập nhật is_active = 0 (Xóa mềm để không làm ảnh hưởng đến các đơn hàng cũ đã tham chiếu ID này).
