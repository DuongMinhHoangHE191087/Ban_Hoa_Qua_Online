Hướng dẫn & Thiết kế Hệ thống Chat Real-time

Tài liệu này cung cấp cái nhìn chi tiết về kiến trúc hệ thống chat real-time của Verdant Market, bao gồm luồng giao tiếp WebSockets, APIs dự phòng (HTTP Fallback), cơ chế cập nhật trạng thái online/offline, tối ưu hóa giao diện (UI/UX), và giải pháp loại bỏ lỗi lặp tin nhắn (Optimistic UI duplication bug).

1. Kiến trúc Hệ thống (System Architecture)

+------------------+ +------------------------+ | Web Browser | <-- WebSocket ---> | ChatEndpoint.java | | (JSP Client JS) | | (Jakarta WebSocket 2.1)| +------------------+ +------------------------+ | | HTTP Fallback | (Gọi Service) | v v +------------------------+ +------------------+ | ChatDeliveryService | | ChatAPI.java | -- (Ghi vào DB) --> | - Lưu DB | | (Jakarta Servlet)| | - Broadcast qua WS | +------------------+ +------------------------+

A. WebSocket Room Management

  • Endpoint: ws://localhost:8080/Ban_Hoa_Qua_Online/ws/chat/{sessionId}
  • Hỗ trợ quản lý phiên hoạt động đa luồng an toàn thông qua ConcurrentHashMap và CopyOnWriteArraySet tại ChatEndpoint.java.
  • Xác thực phiên kết nối dựa trên việc chia sẻ HttpSession thông qua cấu hình HttpSessionConfigurator.class để trích xuất đối tượng User hiện tại một cách an toàn.

B. HTTP Fallback (Cơ chế Dự phòng)

Khi kết nối WebSocket bị gián đoạn hoặc không khả dụng, hệ thống tự động chuyển đổi sang giao thức HTTP POST thông qua servlet ChatAPI.java bằng API action sendMessage. Điều này giúp cuộc trò chuyện không bao giờ bị đứt quãng.

2. Danh sách APIs hỗ trợ (doGet / doPost)

Servlet ChatAPI.java (mapped tại /api/chat) cung cấp các endpoint quan trọng sau:

GET Requests:

  • action=getMessages&sessionId=X: Lấy danh sách tin nhắn cũ trong phiên chat và tự động đánh dấu đã đọc.
  • action=getOnlineStatus&sessionId=X&userId=Y: Kiểm tra trạng thái trực tuyến của người dùng Y trong phòng chat X thông qua kết nối WebSocket thực tế.
  • action=getSessions: Lấy danh sách tất cả các cuộc trò chuyện của người dùng hiện tại (dùng cho việc cập nhật thanh sidebar).
  • action=markRead&sessionId=X: Đánh dấu tất cả tin nhắn trong phiên chat là đã đọc.

POST Requests:

  • action=sendMessage: Gửi tin nhắn qua HTTP (Fallback). Yêu cầu kiểm tra CSRF token qua param _csrf hoặc header X-CSRF-Token.
  • action=createAdminSession: Tạo/tìm kiếm phiên chat hỗ trợ giữa khách hàng và Admin.
  • action=createShopSession&ownerId=Z: Tạo/tìm kiếm phiên chat giữa khách hàng và chủ cửa hàng Z.

3. Tối ưu hóa UI/UX cao cấp (Premium UI/UX Features)

Hệ thống chat được trang bị những tính năng giao diện tiệm cận chất lượng của Facebook Messenger:

  1. Skeleton Loader: Hiển thị các khối tin nhắn giả lập bóng mờ (shimmer animation) trong khi dữ liệu lịch sử đang được tải từ API.
  2. Optimistic Rendering (Hiển thị lạc quan): Tin nhắn được đưa vào giao diện ngay lập tức với trạng thái Đang gửi... (⏳) trước khi máy chủ xác nhận lưu thành công.
  3. Real-time Message Ticks: Trạng thái tin nhắn chuyển động mượt mà:
    • Đang gửi (⏳): Đang đẩy dữ liệu đi.
    • Đã gửi (✓): Máy chủ đã lưu.
    • Đã nhận (✓✓): Đã phân phối tới trình duyệt đối phương.
    • Đã xem (✓✓ màu xanh): Đối phương đang hoạt động trong phòng chat.
  4. Tự động markRead Real-time: Khi người dùng đang mở phòng chat và có tin nhắn mới tới, client tự động kích hoạt API markRead ngầm và ẩn đi thông báo đỏ (unread badge) trên thanh sidebar cho phòng chat đang mở mà không cần tải lại trang.

4. Khắc phục lỗi lặp tin nhắn (Optimistic Duplicate Bug)

Nguyên nhân lỗi: Khi người dùng gửi tin nhắn, client tạo một khối tin nhắn tạm với ID tmp-12345678. Khi WebSocket phát sóng (broadcast) tin nhắn đã lưu từ máy chủ quay trở lại, nó mang ID thật từ cơ sở dữ liệu (ví dụ: 42). Vì 42 khác với ID tạm, client hiểu lầm đó là tin nhắn mới và vẽ thêm một lần nữa.

Giải pháp sửa lỗi: Khi nhận được tin nhắn phát sóng từ WebSocket, client sẽ kiểm tra xem tin nhắn đó có phải của mình không (msg.senderId === currentUserId). Nếu đúng, nó tìm kiếm khối tin tạm thời cũ nhất trong DOM, cập nhật thuộc tính data-msg-id thành ID thật, chuyển biểu tượng trạng thái sang Đã gửi, và dừng quá trình vẽ tin nhắn mới.

ws.onmessage = (e) => {
    try {
        const msg = JSON.parse(e.data);
        if (msg.error) return;
        if (msg.messageId && !renderedIds.has(msg.messageId)) {
            renderedIds.add(msg.messageId);
            const isMine = msg.senderId === currentUserId;
            
            if (isMine) {
                // Tìm tin nhắn tạm thời cũ nhất và nâng cấp nó
                const optimisticEl = chatBox.querySelector('[data-msg-id^="tmp-"]');
                if (optimisticEl) {
                    optimisticEl.dataset.msgId = msg.messageId;
                    setMsgStatus(optimisticEl, 'sent');
                    if (msg.mediaUrl) trackMedia(msg);
                    refreshSidebar();
                    return; // Ngăn chặn duplicate render
                }
            }

            appendMessage(msg, isMine);
            if (!isMine && lastMsgEl) setMsgStatus(lastMsgEl, 'seen');
            if (!isMine) fetch(CTX + '/api/chat?action=markRead&sessionId=' + sessionId).catch(() => {});
            if (msg.mediaUrl) trackMedia(msg);
            refreshSidebar();
        }
    } catch(err) { console.error('WS parse err', err); }
};
